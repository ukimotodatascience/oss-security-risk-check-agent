# OSSセキュリティ診断ルールカタログ

本書は `src/rule_engine.py` で実行されるルール ID の並び、および `src/rules/` 配下の実装と整合するよう整理したカタログです。  
**共通の戻り値型**は「共通返却形式」を参照してください。

# A：コード自体の脆弱性（code）

実装ディレクトリ: `src/rules/A_code/`  
補助実装: `src/rules/A_code/a1_detectors/`（A-1 の言語別検出）

## A-1 OSコマンドインジェクション（Command Injection）

### 概要

外部入力が OS コマンド実行に流れ込み、任意コマンド実行につながるリスクを検知する。

### インプット

- ソースコード（Python、JavaScript / TypeScript、シェルスクリプト等）
- 代表的な抽出対象 API・パターン（例）
  - Python: `os.system`, `os.popen`, `subprocess.run`, `subprocess.Popen`, `subprocess.call`, `asyncio.create_subprocess_shell` 等
  - JS/TS: `child_process` 系、`exec` / `execSync` 等（実装が追跡するトークンに依存）
  - シェル: パイプ、`eval`、外部入力に近い変数展開

### 判定基準

#### 判定条件

1. 外部入力（HTTP クエリ、環境変数、`input()` 等）がコマンド文字列や引数の生成に使われている。
2. かつ、シェル解釈が介在する実行（`shell=True` 相当、シェル経由の one-shot 実行等）が疑われる。
3. 外部入力が配列引数として渡され `shell` が無効な場合は、文脈により注意扱い。
4. 外部入力を用いず固定コマンドのみの場合は問題なしとみなす。

#### 返却値

検知時は `RiskRecord` を生成する。深刻度・メッセージは実装が付与する（`Severity` は `src/models.py` の列挙）。

#### 補足

- `shell=True` やシェル直結 API は強い危険シグナル。
- A-1 は `a1_detectors` により言語別に走査する。

## A-2 SQLインジェクション（SQL Injection）

### 概要

外部入力を含む SQL が安全にパラメータ化されていない場合のリスクを検知する。

### インプット

- ソースコード
- SQL 実行 API、ORM 利用箇所

### 判定基準

#### 判定条件

1. 外部入力を文字列連結・f-string・format で SQL に埋め込んでいる → 高リスク。
2. プレースホルダやパラメータバインドを利用している → 低リスク。
3. テーブル名・カラム名のみ動的など、SQL 構造の一部を動的にしている → 文脈依存の注意。

#### 返却値

検知時は `RiskRecord` を生成する。

---

## A-3 安全でないデシリアライゼーション（Unsafe Deserialization）

### 概要

信頼できない入力を危険な方法でデシリアライズする実装を検知する。

### インプット

- ソースコード
- 利用ライブラリ（例: `pickle`, `yaml.load` の Loader 未指定 等）

### 判定基準

#### 判定条件

1. 信頼できない入力に対し `pickle.loads` / `pickle.load`、危険な `yaml.load` 等 → 高リスク。
2. `yaml.safe_load` 等の安全な API → 問題なし。
3. 危険 API だが入力元が固定かつ改ざん不能と判断できる → 注意。

#### 返却値

検知時は `RiskRecord` を生成する。

## B-1 既知の脆弱性（Known Vulnerabilities）

### 概要

使用中の依存ライブラリが既知の脆弱性情報に該当するかを判定する。

### インプット

- `requirements.txt`, `package.json`, `poetry.lock`, `package-lock.json` 等
- 脆弱性 DB 照合結果（運用で連携する場合）

### 判定基準

#### 判定条件

1. パッケージ名・バージョンが DB と一致し深刻度が高い → 高リスク。
2. 一致するが深刻度が限定的 → 注意。
3. 一致しない → 問題なし。
4. バージョン未固定で照合不能 → 注意。

#### 返却値

検知時は `RiskRecord` を生成する。

# 共通返却形式

ルール実装は `src/models.py` の `RiskRecord` を返す。レポート・JSON 等に落とす際の論理構造の例を示す。

## 返却オブジェクト（例）

```json
{
  "rule_id": "A-1",
  "category": "code",
  "title": "Command Injection",
  "severity": "Medium",
  "file_path": "app/main.py",
  "line": 42,
  "message": "外部入力がシェル経由のコマンド実行に渡される可能性があります"
}
```

### フィールド説明

| フィールド   | 型     | 説明 |
| ------------ | ------ | ---- |
| `rule_id`    | string | 例: `A-1` … `L-7`（`rule_engine.run_all` の実行 ID と一致） |
| `category`   | string | `code`, `dependencies`, `cicd`, `config`, `auth`, `secrets`, `runtime`, `crypto`, `logging`, `maintenance`, `license`, `malware` |
| `title`      | string | ルールの英語タイトル（各 Rule クラスの `title`） |
| `severity`   | string | `Critical`, `High`, `Medium`, `Low`, `Info`（`Severity` 列挙の値） |
| `file_path`  | string | 検知箇所のファイルパス（任意） |
| `line`       | number | 行番号（任意） |
| `message`    | string | 人間向け説明（任意） |

---

## 出典・同期の手引き

| 項目 | パス |
| ---- | ---- |
| ルール実行順・ ID 一覧 | `src/rule_engine.py` の `run_all` |
| メタデータ（クラス名・英語タイトル・カテゴリ） | `docs/.rule_defs.json` |
| 実装本体 | `src/rules/**`（`rule_template.py` の `X-1` は本カタログ対象外） |

旧 `rule_list.md` の「`status` / `confidence` / `evidence` 配列」形式は、現行コードベースの `RiskRecord` とは対応しないため、本ファイルでは `RiskRecord` 形式に置き換えて記載している。
